Исправление дрожания/сдвига flares при приближении в Vulkan (09.07.2026)
=======================================================================

Проблема
--------
Во время рендеринга `VK_DrawSurfList()` переключает `backEnd.or` на
orientation текущего entity. В OpenGL пути в конце
`RB_RenderDrawSurfList()` состояние явно возвращается к мировой
orientation (`backEnd.or = backEnd.viewParms.world`) перед отрисовкой
солнца и flares. В Vulkan пути такого восстановления не было, поэтому
`VK_RenderFlares()` вызывала `RB_AddDlightFlares()` / `RB_AddCoronaFlares()`
с orientation последнего отрисованного entity. `RB_AddFlare()` умножал
мировую позицию источника света на чужую `modelMatrix`, и при определённых
ракурсах/приближении flare смещался относительно реальной лампы.

Дополнительно позиция flare округлялась до целого пикселя в
`R_TransformClipToWindow()`. В OpenGL это не заметно, а в Vulkan, где
размер flare растёт плавно (`size = ... + 8 / -eyeZ`), округление центра
усиливало видимое дрожание.

Исправления
-----------
1. src/vulkan/vk_renderer_draw.c
   - В конце `VK_DrawSurfList()` восстановлены `backEnd.currentEntity` и
     `backEnd.or` к мировой orientation, как в OpenGL.
   - `VK_RenderFlares()` теперь использует `f->windowXF/windowYF` при
     построении flare-квада, устраняя пиксельное дрожание.

2. src/renderer/tr_local.h
   - В `flare_t` добавлены поля `windowXF` и `windowYF` для хранения
     точной субпиксельной позиции центра flare.

3. src/renderer/tr_main.c
   - `R_TransformClipToWindow()` больше не округляет `window[0]` и
     `window[1]` до целых. Теперь она возвращает непрерывные
     viewport-координаты; округление выполняется там, где оно нужно.

4. src/renderer/tr_flares.c
   - `RB_AddFlare()` сохраняет неокруглённые координаты в `windowXF/YF`,
     а округлённые — в `windowX/Y` для совместимости с OpenGL рендером.

Сборка
------
cmake --build build -j$(nproc)

Проверка
--------
Проверено в игре: при приближении к источнику света и повороте камеры flare
больше не сдвигается относительно лампы.


Исправление вертикального flip flares в Vulkan (09.07.2026)
===========================================================

Проблема
--------
В `VK_RenderFlares()` ортографическая матрица для flare-квадов использовала
`vpY` (top-left origin Vulkan viewport) в смещении по Y, хотя сами flare
координаты `f->windowY` имеют bottom-left origin, как в OpenGL. Из-за этого
flares рисовались перевернутыми по вертикали и/или смещенными при
неполноэкранном viewport (например, в portal view или снайперском прицеле).

Исправления
-----------
1. src/vulkan/vk_renderer_draw.c
   - Добавлена переменная `vpYBottom`, равная `backEnd.viewParms.viewportY`.
   - Смещение матрицы по Y (`mvp[13]`) теперь использует `vpYBottom`,
     совпадая с OpenGL `qglOrtho(viewportX, viewportX+width, viewportY,
     viewportY+height, ...)`.

Сборка
------
cmake --build build -j$(nproc)


Исправление volumetric fog (09.07.2026)
======================================

Проблема
--------
Volumetric fog volume pass в Vulkan использовал CLAMP_TO_EDGE для fog texture,
тогда как оригинальный OpenGL использует GL_CLAMP с белым border. Белый border
дает alpha = 1.0 для UV вне [0,1], поэтому вблизи камеры (s < 0) туман
получался полностью непрозрачным. Кроме того, volumetric
fog был отключен по умолчанию (r_vkVolumetricFog 0), хотя в OpenGL volume pass
работает всегда.

Исправления
-----------
1. src/vulkan/shaders/world.frag
   - В volumetric fog volume pass out-of-bounds UV обрабатываются физически
     разумно: близко к камере (s < 0) — нет тумана (alpha = 0), далеко или
     за плоскостью отсечки (s > 1 или t вне [0,1]) — полный туман
     (alpha = 1.0). Внутри текстуры используется сэмплирование fog texture.
   - Сохранена vertex alpha при формировании итогового цвета, чтобы
     соответствовать OpenGL modulate path для неполностью непрозрачного fog.

2. src/vulkan/vk_renderer_draw.c
   - r_vkVolumetricFog теперь включен по умолчанию (1), как и в OpenGL.


Исправление масштабирования fogDistanceVector (09.07.2026)
==========================================================

Проблема
--------
После предыдущих правок volumetric fog в Vulkan всё ещё выглядел заметно
плотнее и непрозрачнее, чем в OpenGL на одной и той же карте.

Причина
-------
OpenGL в RB_CalcFogTexCoords() масштабирует все четыре компонента
fogDistanceVector на fog->tcScale, включая translation-компонент [3]:

    fogDistanceVector[0..3] *= fog->tcScale;

Vulkan использовал VectorScale(), который масштабирует только первые три
компонента. Из-за этого смещение fogDistanceVector[3] оставалось
немасштабированным, координата s fog texture сдвигалась относительно
OpenGL, и плотность тумана получалась другой (в данных сценах — выше).

Исправления
-----------
1. src/vulkan/vk_renderer_draw.c
   - VK_GetFogVectors() теперь явно масштабирует все 4 компонента
     fogDistanceVector на fog->tcScale, совпадая с OpenGL.

2. src/vulkan/shaders/world.frag
   - Убран временный диагностический множитель fogAlpha *= 0.5.

Сборка
------
cmake --build build -j$(nproc)


Изменения в Vulkan backend (08.07.2026)
========================================

Проблема
--------
В Vulkan-рендере lightmap-шейдеры (например, textures/doors/door_c05) выглядели
слишком темными, а distance fog некорректно взаимодействовал с lightmap.

Причина
-------
OpenGL сворачивает пару стадий "lightmap + diffuse (GL_DST_COLOR GL_ZERO)" в
один multitexture-проход функцией CollapseMultitexture(). Distance fog в OpenGL
применяется один раз к результату diffuse * lightmap.

Вulkan не выполнял collapse, потому что CollapseMultitexture() отказывался
работать при qglActiveTextureARB == NULL. В результате шейдер рисовался в
несколько проходов, туман накладывался на каждый проход отдельно, и финальный
цвет получался темнее, чем в OpenGL.

Исправления
-----------
1. src/renderer/tr_shader.c
   - Разрешено сворачивание стадий в Vulkan-режиме (vk_active), даже если
     GL multitexture не инициализирован.

2. src/vulkan/vk_material.c
   - VK_StageDescriptorSet() теперь использует явный lightmap из
     stage->bundle[1], который появляется после collapse.

3. src/vulkan/shaders/world.frag
   - Убран лишний множитель *2.0 у lightmap. Масштабирование уже выполняется
     при загрузке (R_ColorShiftLightingBytes), поэтому дополнительный *2.0
     делал Vulkan ярче OpenGL.
   - Исправлено взаимодействие volumetric fog modulation и distance fog:
     теперь distance fog применяется, даже когда активна volumetric-модуляция.

4. src/vulkan/shaders/world.vert
   - Обновлено условие вычисления vFogFactor для совместимости с новой
     логикой distance/volumetric fog.

5. src/vulkan/vk_renderer_draw.c
   - Удалён отладочный спам [VK_NOFOG] и связанная переменная
     vk_fogDebugCount.

Сборка
------
cmake --build build -j$(nproc)

SPIR-V шейдеры перекомпилируются автоматически и копируются в main/.
